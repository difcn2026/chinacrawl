# 决策记录 · Decision Log

记录项目中的重要架构决策及其上下文，确保后来者（包括未来的自己）理解决策的"为什么"。

## 格式
```
## D-{序号}：{决策标题}
- **日期**：YYYY-MM-DD
- **决策者**：{Codex / 用户名}
- **状态**：提议 / 已采纳 / 已废弃 / 已替代
- **背景**：{为什么需要做这个决策}
- **选项**：
  1. 方案A — 优点/缺点
  2. 方案B — 优点/缺点
- **决策**：{选择了什么}
- **理由**：{为什么选这个}
- **影响**：{这个决策影响了什么}
```

## D-001：采用四层自进化架构
- **日期**：2026-06-03
- **决策者**：Codex（基于用户需求）
- **状态**：已采纳
- **背景**：需要一个系统让 Codex 从"一次性工具"变成"越用越稳的工作系统"
- **选项**：
  1. 四层架构（认知+规则+Agent+记忆）— 结构化强/初期投入大
  2. 单层 AGENTS.md — 简单/无法支撑复杂项目
  3. 完全对话驱动（无文件）— 零成本/不可积累
- **决策**：四层架构
- **理由**：四条价值链路互相增强：认知核心定方向、规则约束执行、Agent 保持上下文干净、记忆让经验复用
- **影响**：项目从此按四层模式运行，所有任务遵守 `.codex/` 下的规则和角色定义

## D-002：短剧流水线采用门禁机制
- **日期**：2026-06-03
- **决策者**：Codex（基于流水线最佳实践）
- **状态**：已采纳
- **背景**：AI 项目容易跳过中间步骤直接生成最终输出，导致质量失控
- **选项**：
  1. 严格门禁（每阶段必须审查通过）— 质量保证/速度慢
  2. 宽松提示（提醒但不强制）— 速度快/容易跳过
- **决策**：严格门禁
- **理由**：内容项目质量一致性 > 速度；AI 生成速度快，审查开销占比较小
- **影响**：`pipeline.rules.md` 定义了 5 个门禁点，`AGENTS.md` 定义了反模式预防跳步
