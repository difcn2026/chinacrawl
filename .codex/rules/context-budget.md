# 上下文预算管理器

> 基于 Feishu 文档「Context Engineering Skills」理念实现
> 配合 AGENTS.md 第七节「258K 硬限制」使用

## 核心原则

```
不猜测，不侥幸——每次重大操作前检查预算。
```

## 预算估算表

| 消耗源 | 单次消耗 (估算) | 说明 |
|--------|----------------|------|
| 一轮对话 | ~2-5K tokens | 含 Agent 上下文 + 响应 |
| 读取一个文件 | ~1-3K tokens | 取决于文件大小 |
| 一次工具调用输出 | ~0.5-2K tokens | Shell/Python 输出 |
| 一个 ComfyUI 生成周期 | ~3-5K tokens | 含提示词 + 结果 |
| AGENTS.md 加载 | ~8-12K tokens | 启动时一次性消耗 |

## 四阶段预算警戒

| 阶段 | 估算消耗 | 剩余空间 | 可执行操作 |
|------|---------|---------|-----------|
| 🟢 冷启动后 | ~15-20K | ~238K | 可推进 2-3 个流水线阶段 |
| 🟡 1-2阶段后 | ~80-120K | ~138K | 建议提前归档，准备交接 |
| 🟠 3-4阶段后 | ~150-180K | ~78K | 必须归档，下一轮对话交接 |
| 🔴 5+阶段后 | ~200K+ | ~58K | 强制新会话 |

## 预算检查命令

在 Agent 执行任何重大操作前，评估当前消耗：

```
1. 自查：本轮对话进行了几轮？
2. 自查：读取了多少文件？
3. 自查：执行了多少次工具调用？
4. 估算：当前总消耗 ≈ 15K (启动) + 轮数×3K + 文件数×2K + 工具调用×1K
5. 判断：剩余空间是否 > 80K？
   - 是 → 继续
   - 否 → 触发归档流程
```

## 上下文压缩策略

当接近 🟡 预警时，优先压缩：

| 压缩方式 | 节省空间 | 代价 |
|---------|---------|------|
| 文件内容摘要替换原文 | 50-80% | 丢失细节，但保留关键信息 |
| 工具输出截断 | 30-50% | 只保留最后几行 |
| 合并已完成步骤 | 20-40% | 用一句话代替多轮对话 |
| 飞书全量归档 | 100% | 新会话需从飞书恢复 |

## 交接文件格式

当需要开启新会话时，生成此文件：

```markdown
# 会话交接 — {日期} {时间}

## 当前项目
- 项目名：{project_name}
- 当前集数：{episode}
- 当前阶段：{stage}

## 已完成
- {completed_item_1}
- {completed_item_2}

## 待执行
- [ ] {todo_1}
- [ ] {todo_2}

## 关键上下文
- 飞书归档链接：{feishu_url}
- 上次决策：{decision}
- 阻塞项：{blocker}

## 新会话启动命令
继续 {project_name} 项目，当前在 {stage} 阶段。
飞书归档：{feishu_url}
```

## 自动化归档触发

以下情况**必须**执行归档：
- 完成一个流水线阶段 (S1→S2, S2→S3, ...)
- 估计上下文消耗超过 150K
- 用户表示要暂停/切换任务
- 每完成一集短剧的全流程
