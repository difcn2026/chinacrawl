# 基础规则 · Base Rules

## 任务生命周期
1. **接收任务** → 理解目标、确认边界、检查是否有歧义
2. **执行任务** → 按规则执行，记录关键决策
3. **自检** → 格式、安全、完整性自查
4. **交付** → 输出文件 + 执行报告
5. **记录** → 写入 observations/ 或 corrections/

## 任务执行规则
- 修改代码前先理解上下文，不确定时先问再动手
- 每次改动保持最小范围，不做无关重构
- 改动后自动验证（有测试则运行，无测试则做快速合理性检查）
- 文件操作优先使用 `apply_patch`，避免全量覆写
- 单一任务不超过 5 个文件改动（超过则拆分子任务）

## 输出规则
- 面向用户的文案使用简体中文
- 技术描述和变量名使用英文
- 生成内容需标注版本号和时间戳
- 输出文件放在任务指定的目录下，不随意创建新目录

## 提交规则
- 不在未经确认时 `git commit` 或 `git push`
- 不在未经确认时创建 PR
- 不在未经确认时删除文件或目录
- 不在未经确认时修改 `.codex/` 架构文件

## 错误处理
- 遇到无法自动修复的错误 → 停止并清晰描述问题 + 建议方案
- 不确定的决策 → 列出选项 + 推荐的默认选择 + 理由
- 不静默跳过错误——要么修复、要么报告、要么标记 TODO（含负责人和时间）

## 禁止行为
- 禁止在未经确认时修改 `.codex/memory/` 之外的配置文件
- 禁止添加版权声明或 license header（除非明确要求）
- 禁止用单字母变量名（循环索引 `i`/`j` 除外）
- 禁止在一条对话中执行互不相关的多个大任务（应拆分）
- 禁止生成无意义的 TODO 注释（TODO 必须含原因 + 负责人 + 日期）
