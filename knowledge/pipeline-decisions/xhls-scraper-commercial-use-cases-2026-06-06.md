## 〇、前置背景

xhls_scraper 是 SearXNG 自建搜索 + Jina AI + trafilatura + Playwright + Ollama 的 11 合 1 Web 数据引擎。对标 Firecrawl，但在中国可用、中文友好、零外部付费依赖。

**核心差异化**: Firecrawl 在中国不可用（被墙 + 无中文搜索优化），国内替代品要么是 GUI 工具（八爪鱼），要么是零散的 Python 脚本。xhls_scraper 一条命令出 markdown，零代码门槛。

## 一、批量抓取 (scrape)

### 能力描述

单页 → markdown，双引擎 (Jina AI + trafilatura) 自动 fallback，支持代理。

### 商业场景矩阵

| 场景 | 具体案例 | 客户画像 | 付费意愿 | 客单价预估 |
|------|---------|---------|:--:|:--:|
| 金融数据聚合 | 批量抓央行/证监会/交易所公告，实时入仓 | 量化基金、投研团队 | 极高 | 5-20万/年 |
| 电商竞品监控 | 京东/天猫/拼多多对手定价、库存、评价 | 品牌电商运营 | 高 | 2-8万/年 |
| 招聘市场情报 | BOSS直聘/猎聘批量职位+薪资 → 行业薪酬报告 | HR SaaS、猎头公司 | 高 | 3-10万/年 |
| 舆情监控 | 微博/知乎/小红书品牌提及批量抓取 | PR/MKT 公司 | 高 | 2-6万/年 |
| 学术/专利检索 | 知网/万方/WIPO 专利批量结构化提取 | 律所、研发中心 | 中 | 1-5万/年 |

### 竞品对比

| 工具 | 优势 | 劣势 |
|------|------|------|
| Firecrawl `/scrape` | 生态成熟，有 SDK | 中国不可用，API 付费 |
| 八爪鱼采集器 | GUI 可视化 | 笨重，不支持命令行/API |
| Python 手写 | 灵活 | 人力成本高，不稳定 |
| **xhls_scraper** | CLI 一条命令，免费，可自托管 | 品牌知名度为零 |

## 二、爬整站 (crawl + download)

### 能力描述

深度递归爬取 (`crawl_site`) + 全站下载为本地 markdown 文件树 (`download_site`)。

### 商业场景矩阵

| 场景 | 具体案例 | 客户画像 | 付费意愿 | 客单价预估 |
|------|---------|---------|:--:|:--:|
| 文档站本地化 | Vue/React/FastAPI 官方文档 → 离线 markdown → RAG | AI 创业公司 | 极高 | 3-15万/年 |
| 知识库建设 | 企业内部 Wiki/Confluence → 结构化知识库 → LLM 问答 | 中大型企业 IT | 极高 | 10-50万/年 |
| 法规合规归档 | 税务/环保/行业法规站全量归档，版本对比 | 律所合规部 | 高 | 5-20万/年 |
| 竞品官网全量分析 | 竞品产品页+定价页+帮助中心完整结构 | 产品/战略团队 | 高 | 3-10万/年 |
| 政府数据采集 | 统计局/海关/市监局公开数据批量下载 | 咨询公司、智库 | 中 | 2-8万/年 |

### 核心卖点

`download_site` 直接输出 `{url}.md` 文件树 → 可直接接 Dify / FastGPT / LangChain 做 RAG。这是 Firecrawl 也没有的"一站到底"体验——竞品需要 scrape → crawl 两步，xhls_scraper 一步到位。

## 三、监控变化 (monitor + monitor-ai)

### 能力描述

- `monitor_page`: 基于 hash 的变化检测
- `monitor_page_ai`: AI Judge 去噪——区分"换了个广告图"和"产品价格真的变了"

### 商业场景矩阵

| 场景 | 具体案例 | 客户画像 | 付费意愿 | 客单价预估 |
|------|---------|---------|:--:|:--:|
| 政策法规追踪 | 证监会/央行/发改委新规发布 → AI 判断实质变更 | 金融机构合规部 | 极高 | 5-30万/年 |
| 竞品动态告警 | 竞品官网价格/功能/团队变动 → 去噪后推飞书/钉钉 | 互联网公司 | 极高 | 3-15万/年 |
| 招标信息监控 | 政府采购网/招标平台新项目即时推送 | 投标公司 | 极高 | 3-20万/年 |
| 供应链风险 | 上游供应商官网公告（停产/涨价/召回） | 制造业采购 | 高 | 5-15万/年 |
| 知识产权保护 | 监控竞品是否抄袭自家商标/文案/设计 | 品牌法务 | 高 | 2-8万/年 |

### 核心壁垒

普通 hash 监控 99% 是噪声（时间戳、广告位、推荐内容变化）。AI Judge 用本地 Ollama 模型做语义对比，过滤掉非实质性变更。这个能力目前**市场上没有竞品**——Firecrawl monitor 也只是 hash 级别，没有 AI 去噪。

## 四、结构化提取 (extract + extract-llm)

### 能力描述

- `extract_structured`: CSS 选择器级提取
- `extract_llm`: Ollama qwen2.5:7b 语义提取，零 API 费用

### 商业场景矩阵

| 场景 | 具体案例 | 客户画像 | 付费意愿 | 客单价预估 |
|------|---------|---------|:--:|:--:|
| 招投标解析 | 招标公告 → 结构化 JSON：项目名/金额/截止日/资质要求 | 投标公司 | 极高 | 5-20万/年 |
| 工商信息提取 | 天眼查/企查查页面 → 股东/法人/注册资本/变更记录 | 投资/尽调 | 高 | 3-10万/年 |
| 财报数据提取 | PDF 财报 → 营收/利润/现金流/同比变化表 | 券商研究 | 高 | 5-15万/年 |
| 房产信息聚合 | 链家/贝壳房源 → 价格/面积/户型/地铁距离 → 比价表 | 房产数据商 | 高 | 3-10万/年 |

### 黑科技

`extract-llm` 用本地 Ollama qwen2.5:7b 做语义提取，不要 API 费用。对比 Firecrawl `/extract` 每次烧 credits，xhls_scraper **零边际成本**。这是商业化定价的杀手锏——可以无限量提供提取服务而不增加成本。

## 六、定价策略建议

### 三档定价（对应三阶段路径）

| 档位 | 价格 | 包含 |
|------|------|------|
| **社区版** (AGPL) | 免费 | 自部署，全部 11 项能力，社区支持 |
| **托管版** (阿里云) | 199/月 | 无需部署，SearXNG + Ollama 托管，API 调用 |
| **企业 Pro** | 5-20万/年 | 私有部署 + SLA + 定制搜索引擎 + AI Judge 调优 + 专属支持 |

### 关键差异化定价点

- **AI Judge 去噪**: 企业 Pro 独占（技术壁垒最高）
- **中文搜索引擎配置**: 托管版起提供（百度/搜狗/必应中国）
- **SLA 保证**: 企业 Pro 99.9% 可用性
- **无限 extract-llm**: 所有档位（边际成本为零的优势）

## 八、Ollama 法律合规分析

> 问题: xhls_scraper 商业化使用 Ollama 是否会有法律风险？
> 结论: 当前架构在四层法规模块上均处于安全区。核心原则——xhls_scraper 是"数据工具"而非"AI 服务"，Ollama 和模型均由用户自行安装。

### 8.1 Ollama 引擎

| 组件 | 许可证 | 商用 |
|------|:--:|:--:|
| Ollama CLI/服务端 | MIT | 完全自由 |
| llama.cpp 依赖 | MIT | 完全自由 |

Ollama 是 MIT 协议，商用零风险。可以打包进商业产品。

### 8.2 模型许可证矩阵

xhls_scraper 默认使用 qwen2.5:7b（通义千问），用户可自行更换。

| 模型 | 许可证 | 中国商用 | 注意 |
|------|--------|:--:|------|
| **qwen2.5** (阿里通义千问) | Apache 2.0 | 安全 | 当前默认，无限制 |
| **deepseek-r1/v3** | MIT | 安全 | 无限制 |
| **Mistral/Mixtral** | Apache 2.0 | 安全 | 无限制 |
| **Phi** (微软) | MIT | 安全 | 无限制 |
| **Yi** (零一万物) | Apache 2.0 | 安全 | 无限制 |
| Llama 3/4 (Meta) | Llama 3 Community | 有条件 | 月活 >7 亿需单独授权 |
| Gemma (Google) | Gemma License | 有条件 | 有禁止用途条款 |

xhls_scraper 默认锁定 qwen2.5:7b（Apache 2.0），用户自换模型的风险归用户侧。

### 8.3 中国 AI 监管法规

| 法规 | 适用范围 | 是否命中 | 分析 |
|------|----------|:--:|------|
| 《生成式人工智能服务管理暂行办法》 | 向中国公众提供生成式 AI 服务 | 不命中 | xhls_scraper 是数据工具，用户本地跑模型 |
| 《互联网信息服务深度合成管理规定》 | 提供深度合成服务 | 不命中 | 不做内容生成，做数据提取 |
| 算法备案要求 | 提供算法推荐服务 | 不命中 | 不涉及推荐算法 |

**关键区分**: xhls_scraper 不是"AI 服务提供商"，是"数据工具"——就像 Photoshop 不会被当成"图像服务提供者"一样。

### 8.4 AGPL 与商业化的兼容性

| 场景 | AGPL 影响 |
|------|-----------|
| 用户下载自部署 | 无限制 |
| 作为 SaaS 提供（托管版） | 允许，纯 SaaS 不算分发 |
| 企业 Pro 私有部署 | 合同约定即可 |
| 有人 fork 后闭源卖 | AGPL 禁止，保护你的商业利益 |

AGPL 实际上保护了商业化——别人不能拿你的代码闭源竞争。

### 8.5 风险评估总结

```
┌────────────────────────────────────────────────────┐
│ 风险层              │ 状态    │ 行动              │
├──────────────────────┼─────────┼───────────────────┤

│ qwen2.5 (Apache 2)  │ 零风险  │ 默认模型没问题    │
│ 中国 AI 监管        │ 零风险  │ 保持"工具"定位   │
│ AGPL 商业化兼容     │ 安全    │ AGPL 保护         │
│ 用户换其他模型      │ 间接    │ 加免责声明 1 条   │
```

### 8.6 建议防御措施

1. **README 免责声明**（一行即可）:
   > "xhls_scraper 是数据提取工具。extract-llm 依赖用户自行安装的 Ollama 模型，用户应确保所使用的模型许可证允许其预期用途。"

2. **默认模型锁定 qwen2.5:7b**（已实现）

**总判**: 商业使用 Ollama + qwen2.5 在法律上完全干净。用户侧换模型的风险通过免责声明隔离。
